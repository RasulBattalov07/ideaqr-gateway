## 3. Успешные флоу (зелёный свет) — с доказательствами

### 3.1 Сценарий 1 — Регистрация и онбординг 🟢

| Шаг | Запрос | Ответ |
|---|---|---|
| Гость провижинится | `POST /api/auth/guest` | `200` · `guest:true`, `trustLevel:10`, `mergeToken` выдан |
| Гость сканирует публичный QR | `POST /api/v2/scan {CAR_TOYOTA_CAMRY}` | `200` · `APPROVED`, **`accessTier:PUBLIC`**, `registrationRequired:true` |
| Регистрация Актора А | `POST /api/auth/register` | `201` · «Регистрация прошла успешно» |
| Вход А | `POST /login` | `200` · `success:true` |
| Новый QR А резолвится | `GET /api/v2/my-qr` | `200` · `identityUid: ba685471…` |
| **Merge гостевой истории → А** | `POST /api/v2/guest/merge` | `200` · **«История гостя перенесена. Записей: 3»** |
| История А содержит гостевой скан | `GET /api/v2/history/me` | `scannedByMe = ['CAR_TOYOTA_CAMRY']` ✓ |

**Вывод:** гостевой вход, тиражируемая публичная проекция карточки (гость видит только
публичные поля), регистрация и **перенос истории гость→гражданин** работают идеально.
Свежий QR гражданина А **резолвится** (доказано в §3.3 — Б сканирует его без ошибки реестра).

### 3.3 Сценарий 3 — P2P Owner-Approval 🟢

| Шаг | Запрос | Ответ |
|---|---|---|
| А сканирует визитку Б | `POST /api/v2/scan {IDENTITY:aaaa…007}` | `200` · **`REVIEW`** «Ожидается подтверждение владельца» |
| А опрашивает ДО согласия | `GET /api/v2/access/{id}/result` | `REVIEW` «Ожидается подтверждение владельца…» |
| Б видит запрос | `GET /api/v2/access/pending` | `[{fromName:"Алия Гость", kind:"PROFILE"}]` ✓ |
| Б подтверждает | `POST /api/v2/access/{id}/confirm` | `200` · `APPROVED` «Владелец подтвердил доступ» |
| А опрашивает ПОСЛЕ согласия | `GET /api/v2/access/{id}/result` | `200` · **`APPROVED` + данные профиля** |
| **Обратный скан (онбординг!)** Б → новый QR А | `POST /api/v2/scan {IDENTITY:ba68…}` | `200` · **`REVIEW` (найдено!)**, не «не найдена» |

**Вывод:** модель «скан = только просмотр, данные открываются лишь после подтверждения
владельца» работает строго. Поллинг на стороне сканирующего отдаёт результат после согласия.
Скан **свежесозданной** личности А успешно резолвится — прямое закрытие риска «не найден».

| Шаг | Запрос | Ответ |
|---|---|---|
| Врач (чистый вход) | `GET /api/auth/me` | `roles:[CITIZEN,DOCTOR]`, `trustLevel:85`, «Врач» |
| **НЕГАТИВ:** личный режим | `POST /api/v2/scan {MED_RX_5521}` | `REJECTED` «доступно только в рабочем режиме» ✓ |
| Включение рабочего режима | `POST /api/v2/mode/work {больница}` | `mode:WORKING`, «Городская больница» |

| **НЕГАТИВ:** 23:00 | `POST /api/v2/scan {MED}` | `REJECTED` «только в рабочее время (08:00–18:00)» ✓ |
| Скан в 10:00 | `POST /api/v2/scan {MED}` | **`REVIEW`** «требует согласия пациента» |
| Пациент Б видит запрос | `GET /api/v2/access/pending` | `[{fromName:"Санжар Доктор", kind:"MEDICAL"}]` ✓ |
| Пациент Б подтверждает | `POST /api/v2/access/{id}/confirm` | `APPROVED` «Пациент подтвердил доступ» |

авторизованного врача). Раскрытая карта содержит полный набор полей (аллергии, хронические
диагнозы, препараты, визиты). Цепочка Врач → Фармацевт замыкается.

## 4. Проваленные / проблемные флоу

Единственная порванная цепочка в линейном мульти-акторном прогоне.

### 🟡 D-1. Спотыкающийся повторный вход после принудительного разлогина (Medium)

**Где рвётся:** Сценарий 2 → Сценарий 4. Админ выдаёт Актору В роль DOCTOR; сервис
корректно завершает активные сессии В (by design, audit 3.9). Дальше В должен войти заново —
и здесь начинается шероховатость.

**Воспроизведено контролируемо** (отдельный изолированный репро, не артефакт harness):

| Шаг | Наблюдение |
|---|---|
| A. Чистый вход врача → `GET /me` | `200` ✓ |
| B. Админ меняет роль → сессии отозваны | `200` «Активные сессии завершены» |
| C. Старая cookie: `GET /api/auth/me` | **`HTTP 200` + plain-text** «This session has been expired (possibly due to multiple concurrent logins…)» — **не JSON 401**; `JSESSIONID` сброшен |
| D. Повторный вход #1 `POST /login` | **`HTTP 403 Forbidden`** (XSRF-cookie тоже инвалидирована) |
| D. `GET /me` после #1 | `401` (не вошёл) |
| E. Повторный вход #2 `POST /login` | `200` ✓ |
| E. `GET /me` после #2 | `200` ✓ — теперь всё работает |
| F. Полностью свежая сессия (чистые cookie) | вход `200`, `/me` `200` — **всегда работает** |

**Две проблемы в одной:**
1. **Нарушение контракта ответа.** Истёкшая сессия отдаёт `HTTP 200` с **текстовым** телом
   вместо задокументированного **JSON `401`**. SPA, делающий `response.json()`, упадёт на
   парсинге либо примет `200` за успех.
2. **Первый повторный вход → `403` (CSRF).** Вместе с сессией инвалидируется и `XSRF-TOKEN`,
   поэтому первый `POST /login` отбивается CSRF-фильтром; вход проходит лишь со **второй**
   попытки (или после того, как клиент заново подтянет XSRF-cookie через GET страницы).

**Влияние:** ровно сценарий из ТЗ — «Админ выдал роль, пользователь должен начать работать».
Пользователь видит неинформативное сообщение про «несколько одновременных входов» (хотя на
деле роль сменил админ) и с первого раза войти не может. Не дыра в безопасности (поведение
fail-closed), но заметная шероховатость онбординга специалиста.

> Примечание по методологии: в самом первом линейном прогоне harness переиспользовал
> «протухшую» cookie, и это совпало с D-1. Я изолировал причину отдельным репро (выше) —
> в свежей сессии (шаг F) и в фокус-прогоне медфлоу **всё проходит на 100%** (§3.4).

## 6. UX-шероховатости (наблюдения «глазами 4 человек»)

1. **🟡 Повторный вход специалиста после смены роли (D-1).** Самая ощутимая. Сообщение
   «несколько одновременных входов» вводит в заблуждение, и первый вход падает в `403`.
   *Рекомендация (без правок сейчас): отдавать JSON `401` на истёкшую сессию и гарантировать
   свежий `XSRF-TOKEN` на ответе `/login`, чтобы вход проходил с первой попытки.*

2. **🟡 Включение специалиста — это 2 раздельных админских действия + ре-логин.** Чтобы
   врач заработал, админ должен (а) подтвердить трудоустройство и (б) выдать профессию, а
   пользователь — перелогиниться. Логично по модели governance, но «тяжело» для онбординга.
   Рабочий режим вдобавок требует **членства в организации** (без него — исключение
   «Рабочий режим недоступен»), т.е. без шага (а) врач не откроет карту даже с ролью.

3. **🟡 Рабочие часы считаются по серверному UTC.** Гейт «08:00–18:00» берёт час из
   `Clock.systemDefaultZone()`, а Render живёт по **UTC** (`serverHour:14` в момент прогона).
   Для казахстанского деплоя (UTC+5) «рабочее время» фактически сместится на **13:00–23:00
   по местному** — латентная локализационная ошибка. Сейчас не мешало (14 UTC = рабочее), но
   на проде для реальных пользователей даст неожиданные отказы/доступы.

4. **🟡 Демо-«машина времени» доступна в проде любому вошедшему.** `POST /api/v2/dev/time
   {hour}` требует лишь аутентификации и подменяет час рабочего гейта для своей сессии.
   Для медицины это не обход (сверху есть согласие пациента + роль + доверие), но для
   **инфраструктуры** (INSPECTOR, где шага согласия нет) вошедший инспектор может
   самовыдать себе доступ вне рабочих часов, выставив `hour=10`. *Рекомендация: закрыть
   `/api/v2/dev/**` профилем dev или ролью админа.*

5. **🟢 Приятные мелочи, которые порадовали.** Гостю на скане честно показывают
   `registrationRequired:true` + CTA «зарегистрируйтесь, чтобы продолжить»; межтенантное
   согласие (врач из тенанта «Больница» спрашивает пациента из публичного тенанта) показывает
   реальное имя запрашивающего; вся история governance (request→decision→interaction→history)
   проставляется на каждом шаге.

## 7. Резюме для стейкхолдеров

- **Готово к демо.** Сквозные сценарии из ТЗ проходят на живом сервере: гость→гражданин с
  переносом истории, админ создаёт/передаёт объекты и выдаёт роли, P2P-подтверждение,
  медицинское согласие с раскрытием карты и рецептом, SOS-эскалация и её закрытие.
- **Безопасность держит границы:** анонимы и граждане не проникают в админ-зону; роли не
  самоназначаются; медкарта защищена и ролью, и согласием пациента.
- **Один реальный дефект (D-1)** — спотыкающийся ре-логин после смены роли админом; не
  блокер, но первым делом стоит сгладить именно его, т.к. он лежит ровно на пути «выдали роль
  специалисту → он начал работать».
- **Два «латентных» риска для реального запуска:** рабочие часы по UTC (локализация) и
  открытая в проде dev-«машина времени» (харднинг инфраструктурного гейта).
