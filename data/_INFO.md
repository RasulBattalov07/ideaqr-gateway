# Папка `data/` — локальная демо-БД

Здесь лежит **runtime-база данных H2** приложения — один бинарный файл `ideaqr.mv.db`.
Он **создаётся автоматически** при первом старте и в репозиторий **не** коммитится
(см. `.gitignore`); отслеживается только этот файл-описание.

## Откуда берутся демо-данные

Это **не** каталог SQL-сидов или дампов. Сами демо-данные заданы **в коде** и
разворачиваются в БД при старте:

| Источник | Что задаёт |
|---|---|
| `src/main/java/.../config/DataSeeder.java` | 3 организации-тенанта, 7 демо-аккаунтов + членства, объект Toyota Camry, личность-визитка «Айдос» |
| `src/main/java/.../service/RegistryClient.java` | Курируемый демо-реестр объектов (Nike, медкарта, услуга, замок AITU, студбилет) — резолвятся при сканировании независимо от тенанта |
| `src/main/resources/db/migration/V1…V8` | Схема БД (Flyway); в prod проверяется на соответствие (`validate`) |

## Предзаполненные демо-сценарии

- **Медицина** — медкарта `MED_RX_5521` открывается врачу/фармацевту в рабочем режиме и
  рабочее время **и** только с согласия пациента (Айдоса).
- **P2P / Owner-Approval** — сканирование визитки `IDENTITY:aaaaaaaa-…-007` шлёт запрос
  владельцу, тот подтверждает — сканеру открывается профиль.
- **Инфраструктура** — умный замок `LOCK_OFFICE_AITU` доступен инспектору по роли
  в рабочем режиме/времени.

Мульти-тенантность: каждая организация (`Городская больница`, `АО «Астана-РЭК»`,
`IDEAQR Retail`) — отдельный изолированный тенант; админ работает кросс-тенантно.

## Сброс к чистому демо

```bash
# останови приложение, затем:
rm -f data/ideaqr.mv.db      # (или удали всю папку data/)
# следующий старт пересоздаст БД начисто и заново засеет демо-данные
```

Демо-аккаунты и cheat-коды объектов перечислены в корневом `README.md`.
