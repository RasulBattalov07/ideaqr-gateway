# IDEA QR DIGITAL GATEWAY — Developer Cheat Sheet 🛠️

> Личная шпаргалка создателя для управления MVP и проведения демо.
> Платформа цифровых взаимодействий: **QR — только точка входа, права решает движок, история неизменяема.**
> Phase 2 — **«Единый национальный QR»**: у человека один личный QR, а раскрываемые данные зависят от **роли и режима сканирующего** (врач → медкарта по согласию, фармацевт → только рецепты, полиция → правовое досье, гражданин → визитка). Онбординг — по номеру телефона через mock-eGov.
> Архитектура **мульти-тенантная** (жёсткая изоляция организаций); предзаполненные демо-сценарии: **Медицина**, **P2P**, **Инфраструктура**, **Право/полиция**, плюс две **трёхсторонние сделки** на том же конвейере: **Диспетчеризация услуг** (заказчик × оператор × исполнитель — сверка личности исполнителя сканом его QR у двери) и **Розничная касса** (покупатель × кассир — корзина/оплата по личному QR, выдача = передача владения при неизменном QR товара).

## ⚡ Быстрый старт

```bash
# Запуск (нужен JDK 17 + Maven). ВНИМАНИЕ: Lombok ломается на JDK 25 — строго JDK 17.
mvn spring-boot:run            # → http://localhost:8080

mvn clean package              # fat jar → target/ideaqr-gateway.jar
mvn clean test                 # весь сьют (105 тестов)
mvn test -Dtest=ИмяКласса      # один класс

# Docker
docker build -t ideaqr-gateway . && docker run -p 8080:8080 ideaqr-gateway
```

- **Сброс БД:** останови приложение и удали папку `./data/` — при следующем старте `DataSeeder` пересоздаст всё (11 аккаунтов, орг-ции, объекты, цифровые досье).
- **H2 console (только локально):** http://localhost:8080/h2-console · JDBC `jdbc:h2:file:./data/ideaqr` · user `sa` · пароль пустой.
- **Prod:** профиль `postgres` (`SPRING_PROFILES_ACTIVE=postgres` + `DB_HOST/DB_PORT/DB_NAME/DB_USERNAME/DB_PASSWORD`).

### 📁 Папка `data/`
Локальная runtime-БД H2 (`data/ideaqr.mv.db`) — **создаётся автоматически** при первом старте; `DataSeeder` заполняет её демо-данными (5 организаций-тенантов, 11 аккаунтов, демо-объекты и товары магазина, личность-визитка, цифровые досье граждан). Это **не** каталог SQL-сидов — сами демо-данные заданы в коде: `DataSeeder.java`, `RegistryClient.java`, `CitizenDossierService.java` и миграции Flyway `db/migration/V1…V10`. Бинарный файл БД в git не попадает; отслеживается только `data/_INFO.md` с описанием для заказчика. **Сброс к чистому демо:** удалить `data/ideaqr.mv.db` и перезапустить.

### 🧪 E2E-проверка
`scripts/e2e-4device.sh` — сквозная симуляция 4 независимых устройств (admin / врач / пациент / новый гражданин) на отдельных cookie-jar: логин, регистрация, сканы всех демо-объектов и визитки. Падает, если хоть одна готовая демо-сущность вернёт «Объект не найден в реестре». Запуск: подними приложение, затем `bash scripts/e2e-4device.sh`.

### 🔗 Золотой Конвейер
Любое действие проходит единую цепочку — обход запрещён архитектурно:
```
Identity → Role → Request → Decision → Interaction → Event → History → Audit
```
`GatewayService.scan()`: резолв объекта → `Request(PENDING)` → `ValidationService` выносит `Decision` (APPROVED/REJECTED/REVIEW) → `Interaction` → многоуровневая видимость (гость = публичная проекция, зарегистрированный = полная) → `Event` + `History` (хэш-цепочка). Отказ = HTTP 200 + `success:false` (это бизнес-вердикт, не ошибка).

**Админ = кросс-тенантный супер-админ** (видит и управляет всеми тенантами). Изоляция тенантов действует для всех НЕ-админов.

## 🚀 Phase 2 — «Единый национальный QR» (Context-Aware Identity)

Бизнес-концепция: **один личный QR на человека**; что он раскрывает — решает контекст сканирующего (**роль × рабочий режим**). Тот же врач вне смены — просто гражданин.

| Кто сканирует личный QR | Что открывается | Механика |
|---|---|---|
| 👤 Гражданин (или специалист в личном режиме) | **Цифровая визитка** сразу; полный профиль — после Owner-Approval | REVIEW + поллинг, как раньше |
| 🩺 Врач (рабочий режим) | **Медкарта** — только после согласия пациента + «выписать рецепт» | контекст-роутинг → консент-конвейер (не обходится!) |
| 💊 Фармацевт (рабочий режим) | **Только рецепты** к выдаче — минимальный доступ по роли | срез без полной карты, без консента (пациент предъявил QR у стойки) |
| 👮 Полицейский (рабочий режим) | **Правовое досье**: несудимость, штрафы, розыск (уровень SECRET) | властное полномочие; каждый просмотр в журнале |
| 💵 Кассир (рабочий режим) | **Чек покупателя**: корзина + «оплачен, не выдан» с итогами — ни профиля, ни истории | гейты кассы: роль CASHIER + доверие ≥ 50 + режим + время |
| 🧰 Исполнитель у двери (у сканирующего активная заявка, назначенная на него) | **SERVICE_VISIT**: «личность подтверждена» + кнопка «Подтвердить приход» / «Подтвердить и оплатить» | приоритет НАД ролевой маршрутизацией; чужой QR контекста заявки не даёт |

- **Умная регистрация (mock-eGov):** пользователь вводит **только номер телефона** → плашка «Это вы? Расул Батталов, ИИН 070420…» → «Да, это я» → автоматически создаются личность, единый QR и **полный пакет данных**: медкарта (`MED-*`), правовое досье (`LEGAL-*`), визитка (`VCARD-*`). Повторный вход — по SMS-коду (**демо-код 1234**). Демо-фишка: любой номер, оканчивающийся на **7**, находит «Расула Батталова».
- **Владелец всегда видит своё:** гражданин открывает СВОЮ медкарту и СВОЮ справку о несудимости без профессиональных ролей (owner-override с записью в журнал).
- **Дашборд из 3 модулей** («Главная»): 🫀 **Медицина и здоровье** (медкарта, рецепты, сценарий Пациент→Врач→Фармацевт) · 🏠 **Услуги и быт** (заявки ЖКХ: вывоз мусора, сантехник — адрес подтягивается из eGov, конвейер `SERVICE_ORDER`) · 🛍 **Бизнес и магазины** (скидочная карта = тот же QR, «мои вещи», сценарии покупки).
- **Новая роль POLICE** (аккаунт `police`, тенант «Департамент полиции Астаны») и **новая категория LEGAL** (классификация SECRET, политика `LEGAL_ACCESS`: роль + доверие ≥ 80 + рабочий режим + рабочее время).
- **Роли трёхсторонних сделок (Flyway V10):** `SERVICE_OPERATOR` (диспетчер: очередь `/queue`, назначение `/assign` — только в рабочем режиме), `EXECUTOR` (универсальный исполнитель: «Мои наряды» `/assigned`), `CASHIER` (кассир: чек по QR покупателя, приём оплаты, выдача) + тип запроса `PURCHASE` (линия чека). Оба подтверждения заказчика (`/arrival`, `/complete`) сверяют отсканированный uid с назначенным исполнителем — чужой QR отклоняется 403.
- Досье-объекты живут в **публичном тенанте** (идентификатор разрешим любому специалисту, доступ к содержимому решает движок политик); для аккаунтов, созданных до Phase 2, досье **доусоздаётся лениво**.

## 🆕 Что нового (product/UX overhaul)

Итог живого демо с заказчиком — оживлены сценарии и убраны «кнопки-пустышки». Без новых таблиц (Flyway `validate` зелёный): рецепты — это governed `Interaction`, ответы на жалобы — уведомления.

- **Нативный скан камерой.** QR зашивает абсолютную ссылку `<base>/s/<id>` — стоковая камера телефона открывает сайт. Не залогинен → панель входа/гостя, после входа скан выполняется автоматически. База: `app.public-base-url` (env `PUBLIC_BASE_URL`), по умолчанию origin запроса.
  > 📱 **Скан с телефона:** `localhost` телефону недоступен — задай `PUBLIC_BASE_URL` (деплой-домен) **или** открой app по LAN-IP ноутбука (`http://192.168.x.x:8080`), тогда QR зашьёт достижимый адрес.
- **Результат скана — отдельная полноэкранная страница** (не левый сайдбар), удобно на мобиле.
- **Врач → Фармацевт.** На медкарте врач «Выписывает рецепт», фармацевт жмёт «Выдать» (`Interaction` PENDING→CONFIRMED).
- **Передача объекта видна получателю** — вкладка **«Мои объекты»** с QR + уведомление новому владельцу.
- **Скан профиля доводится до конца** — после подтверждения владельцем сканирующий видит профиль (поллинг результата).
- **Рабочий режим теперь гейтит** мед/инфра-доступ: в личном режиме — `WORKING_MODE_REQUIRED`.
- **Машина времени** — виджет 🕓 внизу слева (**виден только администратору**, эндпоинт закрыт `ROLE_ADMIN`) задаёт **глобальный демо-час платформы**: выставленное админом время действует на проверку рабочего окна `08:00–18:00` у **всех** пользователей сразу.
- **Жалобы:** админ «Открыть и ответить» → полный текст + смена статуса + ответ пользователю (уведомлением).
- **Единый Уровень доверия (Trust Level 0–100):** одна метрика `Identity.trustLevel` — она и отображается, и гейтит доступ (медицина ≥ 70, инфраструктура ≥ 60). Второй, «геймифицированной» метрики в логике решений нет.
- **Гость:** вкладки истории/жалоб скрыты; история гостя реально переносится в новый профиль при регистрации (read-union по `linkedGuestUids`).
- Фикс логина «с первого раза ошибка» (CSRF-прайминг + ретрай на 403); чип `pharmacist` добавлен на экран входа.

### Сценарии демо «по кнопкам»
0. **Единый QR (флагман Phase 2)** — `citizen` → «Мой QR» → «Демонстрировать». В соседних сессиях сканируют ОДИН И ТОТ ЖЕ QR: `police` (рабочий режим) → правовое досье «НЕ СУДИМ»; `pharmacist` (рабочий режим) → только рецепты + «Выдать»; `doctor` (рабочий режим) → запрос согласия → `citizen` подтверждает → карта + «Выписать рецепт»; `seller` → визитка. Контраст: `police` в личном режиме → тоже только визитка.
0b. **eGov-онбординг** — экран входа → вкладка «eGov · по телефону» → номер `+7 777 777 77 77` → плашка «Это вы? Расул Батталов…» → «Да, это я» → мгновенно готовый профиль с медкартой/досье/визиткой («Главная» → «Мои документы»). Повторный вход: тот же номер + код `1234`.
1. **Guest Conversion** — выйти/гость → скан `RETAIL_NIKE_AF1` → урезанная карточка + кнопка «Зарегистрироваться» → регистрация → та же карточка, но с ценой/отзывами/историей (история гостя сливается; вкладки истории у гостя скрыты).
2. **Owner Approval / P2P** — войти `citizen` → скан `IDENTITY:aaaaaaaa-0000-0000-0000-000000000007` → статус REVIEW, владельцу ушло уведомление. В другой сессии войти `aidos` → «Запросы доступа» → подтвердить → у сканера автоматически открывается профиль владельца.
3. **Врач → Фармацевт** — ⚠️ «Машина времени» управляется **только с аккаунта администратора**: ведущий демо заранее входит как `admin` и виджетом 🕓 ставит глобальный демо-час `10:00` (он действует на всех пользователей платформы). Дальше `doctor`: **рабочий режим** → скан `MED_RX_5521` → ДОСТУП → «+ Выписать рецепт»; затем `pharmacist` (тоже рабочий режим) → скан того же → «Выдать». Негативные ветки: врач в личном режиме → `WORKING_MODE_REQUIRED`; админ ставит 🕓 `22:00` → у врача `OUTSIDE_WORKING_HOURS`; `citizen` → отказ по роли. Врач и фармацевт время не выставляют — они «пожинают» час, заданный админом.
4. **Request→Decision→Interaction** — любой логин, скан `SERVICE_TRASH_PICKUP` → анимированный конвейер с UUID каждого звена.
5. **Передача владельца** — `admin` → «Управление» → `CAR_TOYOTA_CAMRY` → «Передать владельца» → выбрать `seller` (история = `OBJECT_TRANSFERRED`). Войти `seller` → вкладка **«Мои объекты»**: объект с QR + пришло уведомление.
6. **Access Request (инфра)** — `citizen` скан `LOCK_OFFICE_AITU` → ОТКАЗ; `inspector` (рабочий режим + 🕓 рабочее время) → ДОСТУП.
7. **Машина времени** — из-под `admin` (виджет 🕓 доступен только администратору) поставить глобальный демо-час `22:00` → врач/инспектор в **рабочем режиме** сканируют мед-/инфра-объект и получают отказ по времени; админ возвращает `10:00` → у них же ДОСТУП. Смена часа мгновенно действует на все роли — сами врач/инспектор менять время не могут.
8. **Жалоба + ответ** — `citizen` после скана → вкладка «Жалобы» → подать; `admin` → «Жалобы» → «Открыть и ответить» → статус «Решена» + текст → пользователю приходит уведомление.
9. **Услуги и быт — три роли** — `citizen` → «Услуги и быт» → заказать «Вывоз мусора» (статус «Новая · ждёт оператора»). `operator` (**рабочий режим**) → «Услуги и быт» → плашка «Диспетчерская» → «Назначить исполнителя» (Арман Бекетов) → у заказчика в реальном времени появляется **карточка исполнителя**, у `executor` (рабочий режим) — наряд. «Приход»: `executor` → «Мой QR» → «Демонстрировать»; `citizen` сканирует → «личность сверена» → **«Подтвердить приход»** → у всех трёх сторон «В работе». После «работы» `citizen` сканирует QR исполнителя ещё раз → **«Подтвердить и оплатить»** → у всех «Оплачено · завершена». Чужой QR у двери кнопки не даст — это и есть проверка личности.
10. **Магазин и касса** — `citizen` → «Бизнес и магазины» → скан `SHOP_TSHIRT_UNIQLO` → **«Оплатить на месте»** (товар «оплачен, но не выдан»), а `SHOP_JEANS_LEVIS` и `SHOP_SNEAKERS_NB` → **«В корзину»**. У кассы: `cashier` (**рабочий режим**) сканирует **личный QR покупателя** → чек: корзина с суммой + «оплачен · не выдан». «Не хватает средств»: `citizen` тут же убирает джинсы из корзины («Бизнес и магазины» → Корзина → «Убрать») — чек кассира обновляется сам. Кассир «Принять оплату», затем «Выдать» по каждой позиции → владение переходит покупателю (**QR товара не меняется**), вещь появляется в «Мои вещи», след `OBJECT_TRANSFERRED` — в журнале обоих.
