## 0. Вердикт за 30 секунд (TL;DR для инвест-комитета)

Дизайн — авторская дизайн-система fintech-класса, не бутстрап-шаблон.

**НО.** Между «это можно показать инвестору» и «это можно продать B2B-клиенту завтра» — пропасть, и в неё проваливаются три вещи: (1) **критическая ошибка деплоя** — прод поднят НЕ в защищённом профиле, из-за чего публично открыта консоль базы данных и нет CSP; (2) **кладбище мёртвых эндпоинтов**, часть из которых прямо сейчас отдаёт **HTTP 500**; (3) продукт всё ещё **демо**, а не сервис — нет онбординга организаций, биллинга, SSO, интеграций и реальных данных.

# 1. THE GOOD — что сделано на мировом уровне

Это не комплименты. Каждый пункт я подтвердил живым вызовом прода.

### 1.1. Governed Access — настоящий серверный движок решений, а не `if` на фронте 🏆
Полный путь «Личность → Запрос → Решение → Действие → История» исполняется на сервере, и решение зависит от контекста. Один и тот же скан даёт разный вердикт в зависимости от роли, категории объекта, доверия, режима и времени. Живые ответы прода:

```
citizen → CAR_TOYOTA_CAMRY → APPROVED (tier=FULL)  "Объект общедоступен"  + полный payload (цена, отзывы, альтернативы)
citizen → MED_RX_5521      → REJECTED (HIGH)        "...разрешён только врачам и фармацевтам"
citizen → TOTALLY_FAKE_999 → REJECTED (MEDIUM)      "Объект не найден в реестре"
```
Каждый ответ возвращает **полную цепочку UUID** (`requestUid/decisionUid/interactionUid/historyUid`) — это честный governed access (`GatewayService.scan` → `ValidationService.coreDecide`, `ValidationService.java:94`). Для технического покупателя это главный сигнал: здесь есть **продуктовое ядро**, а не лендинг.

### 1.2. Согласие пациента на медкарту — это killer-feature, и она работает 🏆🏆
Самое сильное, что есть в продукте. Прохождение профессиональных гейтов (роль + доверие + рабочий режим + часы) **необходимо, но НЕ достаточно**, чтобы открыть медкарту. Живой прогон от лица врача:

```
doctor (личный режим)  → MED_RX_5521 → REJECTED  "доступна только в рабочем режиме"
doctor → mode/work (Городская больница) → mode=WORKING, activeRole=DOCTOR
doctor (рабочий режим) → MED_RX_5521 → REVIEW    "Доступ к медкарте требует согласия пациента"
                                                  ← аллергии и диагнозы в payload ОТСУТСТВУЮТ
```
То есть **даже полностью авторизованный врач не видит карту без явного согласия пациента** (`GatewayService.java:114`). Это закрывает «парадокс согласия» (раньше визитка спрашивала разрешение, а медкарта — нет) и попадает ровно в мировой тренд **consent management / privacy-by-design**. В мире это продают как отдельный продукт (consent ledger). Здесь оно встроено в ядро и демонстрируемо.

### 1.3. Минимизация данных (data minimization) — сервер реально режет payload 🏆
Гость и гражданин получают **одинаковое решение** (APPROVED), но **разный объём данных**. Это не «спрятали на фронте» — это вырезано на сервере белым списком (`PublicCard.project`). Живое сравнение одного и того же товара:

```
citizen → RETAIL_NIKE_AF1: {productName, brand, rating, description, PRICE, reviews, alternatives, loyalty}
guest   → RETAIL_NIKE_AF1: {productName, brand, rating, description}   ← price/reviews/alternatives/loyalty ВЫРЕЗАНЫ
                            tier=PUBLIC, registrationRequired=true, cta="...необходимо зарегистрироваться"
```
Честная реализация GDPR-style минимизации + встроенная воронка конверсии гостя. Сильно.

### 1.4. Неизменяемый аудит с live-верификацией хэш-цепочки 🏆
«Неизменяемость» здесь не декларируется, а **доказывается**. Админ-эндпоинт реально пересчитывает хэш-цепочку журнала:
```
admin → GET /api/admin/audit/verify → {"valid":true,"entriesChecked":18}
```
Append-only журнал с хэш-сцеплением (`AdminController.java:136`) — это именно то, что покупают регуляторно-чувствительные B2B/B2G клиенты (медицина, госсектор, инфраструктура). Это потенциальный **ров (moat)**, а не фича.

### 1.5. SOS → реальная очередь у администратора (бывший «театр» закрыт)
В прошлом аудите SOS был пустышкой (уведомление улетало самому отправителю, `Workflow` никто не читал). Теперь — проверено вживую: гражданин жмёт SOS, и запись **реально появляется у админа, кросс-тенант**:
```
citizen → POST /api/v2/sos {"message":"Пожар в здании"}
admin   → GET /api/admin/sos → [{"fromName":"Дамир Оспанов","message":"SOS: Пожар в здании","status":"PENDING"}]
admin может закрыть: POST /api/admin/sos/{id}/resolve
```
`Workflow(SOS_ESCALATION)` теперь потребляется, а не пишется в стол (`AdminController.java:153`).

### 1.6. Рабочий режим и «машина времени» — честные серверные рычаги для демо
Переключатель «Личный/Рабочий» реально открывает профессиональные категории, а час сессии мокается **на сервере** (нельзя подделать с клиента, `ScanRequest` намеренно не несёт время — `ValidationService.java:111`):
```
POST /api/v2/dev/time {hour:10} → {effectiveHour:10, workingHours:true}   (serverHour был 8)
```
Это лучшее «демо-оружие»: на сцене можно вживую показать «днём можно — ночью нельзя».

### 1.7. Зрелая для стадии безопасность (подтверждено вживую)
- **CSRF** включён, double-submit, с **ротацией токена при логине** (подмена токена → `403`).
- **Сессионная кука закалена**: `JSESSIONID … Secure; HttpOnly; SameSite=Lax`.
- **BCrypt**, метод-уровневые `@PreAuthorize` поверх URL-матчера, **тенант-изоляция** (чужие личности не резолвятся), форс-смена пароля, rate-limit фильтр.
- **Защита вывода на фронте дисциплинирована**: `esc()` — настоящий HTML-экранировщик (`&<>"'`), плюс `safeUrl()` блокирует `javascript:`/`data:` ссылки (`app.js:38`, `app.js:47`). Атакующие поля (`objectUid`, текст SOS, имена, жалобы, уведомления) обёрнуты в `esc()` — выборочно проверил, дисциплина соблюдается.
- HTTP-заголовки безопасности присутствуют: HSTS, `X-Content-Type-Options: nosniff`, `Referrer-Policy`.

### 1.8. UI/UX — это авторская дизайн-система fintech-класса, а не шаблон
Честный ответ на вопрос «дорогой enterprise или поделка»: **ближе к дорогому**, и это видно в коде, а не на скриншоте:
- **43 дизайн-токена** (CSS custom properties), темизация **тёмная + светлая** через подмену токенов (вся библиотека компонентов переключается без правок per-component).
- **Премиальная типографика, self-hosted**: Manrope (display) · Golos Text (кириллица) · JetBrains Mono (коды) — не системные дефолты; работает в закрытых сетях без утечки на CDN.
- Продуманная визуальная идентичность: «deep-navy fintech с казахским характером», стеклянные панели, золотой акцент «доверия», ненавязчивый орнамент кошкар-мюйз (inline SVG, CSP-safe).
- **Доступность**: модалки с focus-trap вместо нативных `prompt/alert`, `aria-live` тосты, `prefers-reduced-motion`, ARIA-атрибуты, честный баннер «демонстрационная версия» (не выдаёт себя за госсистему).
- Админ-панель реально управляет: блок/разблок, смена роли, выдача/снятие админа с завершением сессий, сброс пароля с одноразовым показом, ответ на жалобу → уведомление автору, **передача владельца объекта с сохранением истории**.

# 2. THE BAD — недочёты: баги, кривой UX, логические нестыковки

Severity: 🔴 блокер · 🟠 серьёзно · 🟡 косметика/полировка.

### 2.1. 🔴 КРИТИЧНО: прод поднят НЕ в защищённом профиле — открыта консоль БД
Это findings №1, и технический Due Diligence находит его за 5 минут. Факты с живого сервера:
```
GET /h2-console/         → 200   ← веб-консоль H2 ПУБЛИЧНО доступна на проде
заголовки на / и /api/health: X-Frame-Options: SAMEORIGIN  (а не DENY)
                               Content-Security-Policy: ОТСУТСТВУЕТ
```
В коде CSP и `frame DENY` включаются **только когда H2-консоль выключена** (`SecurityConfig.java:107–121`). Раз на проде мы видим `SAMEORIGIN` + отсутствие CSP + живую `/h2-console/` → **прод работает с `spring.h2.console.enabled=true`, то есть в дефолтном профиле, а не в `prod`**. Это противоречит `CLAUDE.md` («prod profile … H2 console hard-disabled … strict CSP»).

Чем это грозит:
- **Публично открытая консоль БД.** В дефолтном профиле БД — файловая H2, пользователь `sa`, пустой пароль. Кто дойдёт до консоли и знает/угадает JDBC URL — читает и **меняет всю базу**: BCrypt-хэши, персональные данные, и (главное для вашего нарратива) **может подделать «неизменяемый» аудит**. Я это специально **не** эксплуатировал, но факт доступности — критичен.
- **Нет CSP** → исчезает второй слой защиты от XSS/clickjacking. Сейчас единственный барьер — дисциплина `esc()` на фронте; стоит ей где-то дать сбой (см. 2.4) — и ловить нечем.
- **`ddl-auto=update` вместо `validate`** → схема не валидируется Flyway, риск тихого дрейфа.

**Хорошая новость:** это не баг кода, а одна строка деплоя. `SPRING_PROFILES_ACTIVE=prod` на Render — и сразу зеленеют: H2-консоль выключена, `validate`, требуется `H2_PASSWORD`, появляется CSP и `frame DENY`. **Сделать до любого показа покупателю/пилота.**

### 2.2. 🟠 Пять «Document-22» эндпоинтов отдают HTTP 500 на проде
Аутентифицированный гражданин дёргает живой API — и половина «каталога Document-22» падает:
```
GET /api/v2/foundation     → 500   GET /api/v2/consents      → 500
GET /api/v2/policies       → 500   GET /api/v2/delegations   → 500
                                    GET /api/v2/relationships → 500
```
Это не «не реализовано» — это **смонтировано и публично крашится**. Технический покупатель, который «потыкает» API, найдёт это мгновенно. (Плюс: эти ручки нигде не вызываются фронтом — см. Раздел 3.) Хотя бы стектрейсы наружу не текут — generic-обработчик отдаёт чистый русский текст.

### 2.3. 🟠 Незнакомые маршруты отдают 500 (а не 404), `/zzz` → 401
```
GET /api/admin/zzz → 500       GET /login (GET) → 500       GET /zzz (любая страница) → 401
```
Несуществующие пути валятся в generic-500 вместо 404/405, а неизвестная не-API страница даёт 401 вместо отдачи SPA-оболочки/404. Для DD это «неаккуратный краевой контур»; для SPA с deep-link — потенциально сломанные прямые ссылки.

### 2.4. 🟡 Нет CSP + точечные неэкранированные интерполяции = латентный риск XSS
Сам по себе `esc()` хорош, но в `app.js` есть интерполяции **без** `esc()` (заголовки вердиктов `${v.title}`, подзаголовки карточек через `cardHead(... d.brand ... d.sku ...)`). Сегодня туда попадают серверно-курируемые данные, риск низкий. Но при **создании пользователем объектов/QR** (`/api/admin/qr/create`, `displayName`) эти поля могут стать атакуемыми, а **CSP на проде нет** (см. 2.1) — то есть страховки второго уровня нет. До B2B: восстановить CSP + прогнать все innerHTML-синки через `esc()`.

### 2.5. 🟡 `/api/health` рапортует `"stage":3`, а документация — Stage 4
```
GET /api/health → {"success":true,"message":"...работает","details":{"stage":3,"status":"UP"}}
```
Мелочь, но для DD это «карта не сходится с территорией»: Stage-4-фичи (сессии, организации, SOS, гостевые личности) реально задеплоены, а health-эндпоинт честно говорит «stage 3».

### 2.6. 🟡 SOS: текст «администраторы оповещены» всё ещё слегка обещает больше, чем делает
Ответ SOS: `"...Приоритет повышен, администраторы оповещены."` Фактически админам **не уходит push/уведомление** — создаётся `Workflow`-строка, которую админ увидит, **только если сам откроет вкладку «Тревоги»**. Уведомление по-прежнему пишется **самому отправителю** (`GatewayService.java:329`). Очередь — реальная (это уже хорошо), но «оповещены» подразумевает доставку. Либо завести реальное уведомление админам, либо смягчить текст до «зарегистрировано, видно администраторам в панели».

### 2.7. 🟡 Флагманский флоу согласия требует ручного обновления у пациента
Врач результат **поллит**, а пациент видит входящий запрос только после перерисовки (клик по вкладке / F5) — на фронте нет поллинга `/api/v2/access/pending` со стороны пациента. На соло-демо это «шероховатость хореографии»: после скана врачом надо вручную обновить терминал пациента. Не баг, но единственная заметная зазубрина на самом красивом сценарии. Добавить лёгкий поллинг — и флоу станет бесшовным.

### 2.8. 🟡 Имя владельца в P2P-запросе теряется
При сканировании визитки врачом запрос показывает владельца как «Пользователь» вместо «Айдос Серіков» (`displayName` не нашёл `User` по `identityUid` бизнес-карты). Либо баг связки `User↔Identity` для seed-карты, либо намеренная анонимизация до согласия — но выглядит как недоделка.

### 2.9. 🟡 Render free-tier: холодный старт
Первый запрос после простоя — это спин-ап (десятки секунд «мёртвой» загрузки), дальше ~0.5с. Для демо инвестору/клиенту — мина: прогревайте инстанс заранее или поднимайте платный.

# 3. THE UGLY — что надо безжалостно вырезать или переделать

Это фичи/поверхности, которые **дешевят продукт**: они существуют, но либо мертвы, либо путают, либо противоречат собственному нарративу. M&A-покупатель воспринимает мёртвый код как «технический долг и неаккуратность», а не как «задел».

### 3.1. 🔪 Кладбище мёртвых эндпоинтов — обрезать или достроить
Бэкенд монтирует ручки, которые **фронт никогда не вызывает** (сверка `@RequestMapping` ↔ все `fetch` в `app.js`):

| Эндпоинт | Состояние | Вызывается в UI? | Приговор |
|---|---|---|---|
| `/api/v2/foundation` | **500** | нет | вырезать |
| `/api/v2/policies` | **500** | нет | вырезать или достроить Policy-движок |
| `/api/v2/consents` | **500** | нет | вырезать или связать с согласиями |
| `/api/v2/delegations` | **500** | нет | вырезать |
| `/api/v2/relationships` | **500** | нет | вырезать |
| `/api/v2/wallet` | 200, но **мёртвый** | нет (UI зовёт `/my-qr`,`/my-objects`) | вырезать |
| `/api/admin/events` | 200, работает | нет (нет вкладки) | либо вкладка, либо вырезать |

Это **расширение поверхности атаки и поддержки без продуктовой отдачи**, плюс пять публичных 500-ответов. Сигнал инвестору — «построили модель данных Document-22, но не довели до продукта». Решение бинарное: либо завернуть в когерентную фичу, либо убрать из публичного API.

### 3.2. 🔪 `trustScore` воскрес в `/api/v2/wallet`
Был большой и правильный рефактор: «доверие — одна метрика» (`trustLevel`), а геймифицированный `trustScore` убрали из `/api/auth/me`. Но чистка **неполная** — `wallet` всё ещё отдаёт оба:
```
GET /api/v2/wallet → {... "trustScore":50, "trustLevel":50 ...}
```
Плюс `DataSeeder` всё ещё проставляет `.trustScore(78)`, а в `Identity` живёт поле `trustScore`. Это ровно та двусмысленность «два числа 0–100», которую вы убирали с витрины. Добить: убрать `trustScore` из DTO/энтити/сидера, раз эндпоинт всё равно мёртвый — вырезать целиком.

### 3.3. 🔪 Ручной ввод «внутренних» идентификаторов учит неправильному
Плейсхолдеры `CAR_TOYOTA_CAMRY` и быстрые чипы `RETAIL_NIKE_AF1`, `MED_RX_5521` отлично смотрятся на сцене, но **в реальном продукте пользователь никогда не печатает UID — он сканирует.** Ручной ввод приучает к «внутренней кухне» и дешевит продукт в глазах того, кто ждёт consumer-grade. Подписать блок как «демо-ввод» или спрятать за демо-флагом.

### 3.4. 🔪 «Машина времени» 🕓 висит у всех в проде
Плавающий виджет мока времени виден **любому** залогиненному, включая обычного гражданина, которому он не нужен и непонятен. Это мощный демо-инструмент, но в проде у всех — выглядит как недоубранная отладка. Спрятать за демо-флагом / показывать только админу.

### 3.5. 🔪 Конверсия гостя «разлогинивает»
CTA после гостевого скана ведёт в регистрацию через `doLogout()` → экран входа. Пользователь нажал «зарегистрироваться», а его будто «выкинуло», да ещё и потерян контекст только что показанной карточки. Это анти-паттерн воронки. Вести в регистрацию без видимого «логаута» и возвращать к карточке после.

# 4. INVESTOR'S VERDICT — стоимость, готовность и путь к единорогу

### 4.1. Честная оценка стоимости «как есть» (без розовых очков)
Вы просили правду как у покупателя, который берёт стартапы за сотни миллионов. Правда такая: **сегодня это не актив за сотни миллионов. Это сильный pre-seed прототип.**

Что есть: отличное инженерное ядро, редкая для стадии безопасность, авторский UI, и — главное — **правильный продуктовый тезис** (consent + governed access + immutable audit). Что покупает M&A за сотни миллионов: **выручку, контракты, клиентов-логотипы, ров и команду.** Здесь пока: ноль выручки, ноль платящих клиентов, демо-данные (7 сидов), один инстанс free-tier, один критический мисконфиг.

| Сценарий | Честный диапазон | Чем обоснован |
|---|---|---|
| **Сегодня, acqui-hire / IP-сделка** | низкие 5–6 значных $ | качество кода + IP + основатель, но без traction |
| **Pre-seed раунд сейчас** | оценка **$0.7M–$3M** (привлечение ~$100–500k) | рабочий govtech/accesstech-прототип, KZ-контекст, ясный wedge |
| **После Phase 2 + 1–2 пилота-логотипа (AITU)** | **$5M–$15M** seed | повторяемый онбординг + первая выручка + референс |
| **«Единорог»** | требует Phase 3: выручка $X0M ARR, мульти-страна, ров на комплаенсе | см. 4.4 |

Готовность: **демо-инвестору ~ готов на 90%** (можно показывать, happy-path всех ролей работает вживую). **B2B-прод ~ 25%** (нет инфраструктуры, онбординга, биллинга, SSO, интеграций, реальных данных, и есть P0-мисконфиг). Это нормально для стадии — но не называйте это «готово к масштабированию до национального уровня». Пока — нет.

### 4.2. Чего критически не хватает, чтобы продать B2B «завтра»
1. **Самостоятельный онбординг организации** — сейчас тенанты/специалисты заводятся только сидером/админом. B2B-клиент должен сам завести оргструктуру, пригласить сотрудников, раздать роли.
2. **SSO / SCIM (SAML, OIDC)** — ни один enterprise/вуз не заведёт сотни аккаунтов вручную. Это вход №1 в B2B.
3. **Биллинг и тарифы** — нет подписок, лимитов, метрик потребления. Без этого нет бизнеса, только демо.
4. **Реальные данные вместо реестра-заглушки** — `RegistryClient` отдаёт 6 захардкоженных объектов. Нужны импорт/API/интеграции.
5. **Прод-инфраструктура** — PostgreSQL-профиль уже в коде, но прод крутит файловую H2 на free-tier. Нужны управляемая БД, бэкапы, секреты, observability, CI/CD, SLA, домен.
6. **Экспорт аудита / отчётность** — «неизменяемый аудит» надо уметь выгружать (PDF/CSV/SIEM) — это то, за что платит комплаенс.
7. **Мобильный клиент** — продукт про QR обязан быть phone-first (PWA или нативка с камерой), а не desktop-SPA с ручным вводом UID.

### 4.3. P0 — что починить НЕМЕДЛЕННО (до любого показа покупателю)
1. 🔴 **`SPRING_PROFILES_ACTIVE=prod`** на Render → закрыть H2-консоль, включить CSP + `frame DENY`, `validate`, требовать `H2_PASSWORD`. **Одна строка, чинит несколько findings разом** (2.1).
2. 🔴 **Убить/починить 5 эндпоинтов, отдающих 500** (3.1) — публичные краши в API недопустимы на DD.
3. 🟠 Перевести прод на **PostgreSQL** + платный инстанс (убрать холодный старт и файловую H2).

5. 🟡 Поправить `health.stage`, текст SOS, вырезать `trustScore` из `wallet`, добить мёртвые ручки.

### 4.4. Phase 2 → траектория единорога (что строить и в каком порядке)
Тезис позиционирования: **перестать называть это «QR-приложением». Это «слой управления доступом и согласием с неизменяемым аудитом» (Consent & Access Governance Layer)** — региональный гибрид Okta-для-физического-доступа + consent-ledger. QR — лишь идентификатор на входе.

1. **Фундамент (0–2 мес):** P0 выше + PostgreSQL + CI/CD + observability + бэкапы + домен + восстановление пароля по email. Сделать продукт **эксплуатируемым**.
2. **Wedge — кампус AITU (2–5 мес):** один платящий логотип. Доступ в здания/лаборатории, студенческий билет, библиотека, экзамены, инфраструктурные замки (`LOCK_OFFICE_AITU` уже в демо — это не случайно правильный выбор ниши). Land-and-expand на 2–3 вуза. **Это ваш самый быстрый путь к первой выручке и референсу.**
3. **Платформизация (3–8 мес):** self-service онбординг организаций, white-label, биллинг/подписки, роли per-org, **SSO (SAML/OIDC) + SCIM**, выгрузка аудита (SIEM/PDF).
4. **Интеграции = ров (6–12 мес):** медицина — EMR/**FHIR**; идентичность — нац. ID / eGov; вузы — SIS; инфраструктура — IoT-замки. Каждая интеграция превращает демо-категорию в реальный рынок.
5. **Mobile-first (parallel):** PWA/нативка, камера-центричный UX, офлайн-кэш, push (в т.ч. для пациента и для админ-SOS — закрывает 2.6/2.7).
6. **Комплаенс как продукт (6–18 мес):** соответствие закону РК о перс. данных + **резидентность данных**, дорожка к ISO 27001 / SOC 2. Consent-ledger + immutable audit — продавать **как фичу комплаенса**, а не как технический винтик. Это и есть ваш ров.
7. **Ре-платформа фронта (тех. долг):** монолитный `app.js` (2757 строк vanilla JS) — это виртуозная, но не масштабируемая на команду история. До найма фронт-команды — переезд на компонентный фреймворк + TypeScript + тесты. Сейчас на фронте **нет тестов**; для «национального уровня» это блокер качества.

### 4.5. Резюме одной фразой
**Ядро — на уровень выше типичного MVP и бьёт точно в правильный тренд (consent + governed access + immutable audit). Но прод выдаёт инженерную незрелость (открытая консоль БД, 500-эндпоинты, демо-данные), а продукта-как-сервиса (онбординг, биллинг, SSO, интеграции) ещё нет.** Это **финансируемый pre-seed с сильной командой и правильным wedge (вузы/govtech KZ)**, а не готовый к поглощению за сотни миллионов актив. Закройте P0 за неделю, возьмите пилот AITU за квартал — и разговор о деньгах сменится с «прототип» на «seed с traction».
