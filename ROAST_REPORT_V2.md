# 🔴 CRITICAL

## C-1. Rate-limiting обходится подменой `X-Forwarded-For` → брутфорс паролей + раздувание БД (DoS)

**Где:** `config/RateLimitingFilter.java:134-141` (`clientIp`), `:90-98` (login/guest/register keyed by IP), `:111-113` (`buckets.clear()`).

**Что не так:**
```java
private String clientIp(HttpServletRequest request) {
    String forwarded = request.getHeader("X-Forwarded-For");
    if (forwarded != null && !forwarded.isBlank()) {
        int comma = forwarded.indexOf(',');
        return (comma > 0 ? forwarded.substring(0, comma) : forwarded).trim(); // ← берётся ЛЕВЫЙ, т.е. КЛИЕНТСКИЙ
    }
    return request.getRemoteAddr();
}
```
Лимитер ключует «жёсткие» публичные ведра (`/login`, `/api/auth/guest`, `/api/auth/register`) по `X-Forwarded-For`. Этот заголовок **полностью контролируется клиентом**: на Render/Heroku прокси лишь дописывает реальный IP справа, а берётся **левое** значение. Меняя заголовок на каждый запрос (`X-Forwarded-For: 1.2.3.4`, `1.2.3.5`, …), атакующий получает **новое ведро на каждый запрос** → лимит не действует вообще.

**Импакт (тройной):**
1. **Брутфорс паролей** (audit 4.9 объявлен «починенным» — на деле нет): `loginCapacity=10/мин` обходится → неограниченный перебор паролей демо-аккаунтов (`Admin123!`, `Doctor123!` и т.п. документированы в README).
2. **Раздувание БД / resource-exhaustion DoS:** каждый `POST /api/auth/guest` создаёт **5+ строк** (`Identity` + `Qr` + `Assignment` + `User` + ≥2 `History`) — см. `UserService.createGuestAccount:128-153`. Без лимита это бесконечный рост на 1-ГБ диске (`render.yaml:46`) до отказа.
3. **Усилитель:** `if (buckets.size() > MAX_TRACKED_KEYS) buckets.clear();` (`:111`) — наводнив фильтр 50 000 подделанных ключей, атакующий **обнуляет ведра ВСЕХ пользователей**, включая счётчики брутфорса. Fail-open by design.

**Как воспроизвести:** `for i in $(seq 1 100000); do curl -H "X-Forwarded-For: 10.0.$((i/256)).$((i%256))" -X POST .../login -d 'username=admin&password=...'; done` — 429 не будет.

**Почему CRITICAL:** один заголовок полностью отключает все три публичных защитных контура, которые проект подаёт как реализованные.

# 🟠 HIGH

## H-1. «Неизменяемый» хеш-чейн аудита РВЁТСЯ при конкурентных запросах (и невозможен на нескольких репликах)

**Где:** `service/AuditService.java:56-84` (`@Transactional synchronized record(...)`), `:127-144` (`verifyChain`).

**Что не так:** метод `record` помечен и `@Transactional`, и `synchronized`. Spring оборачивает транзакцию **снаружи** прокси, а `synchronized` — это код **внутри** целевого бина. Порядок: `begin tx → [lock] target.record() → [unlock] → commit tx`. То есть **блокировка снимается ДО коммита**. При `READ_COMMITTED` (дефолт):
- Поток T1: читает `tip = X`, считает `prevHash = hash(X)`, `save` (ещё не закоммичено), снимает lock.
- Поток T2: берёт lock, читает `tip` — **строку T1 ещё не видит** → тоже `prevHash = hash(X)`.
- Получаются **две записи с одинаковым `prevHash`** → `verifyChain()` на втором звене вернёт `valid=false`.

**Импакт:** достаточно **двух одновременных сканирований** (`GatewayService.scan → auditService.record`) разными пользователями, чтобы «неизменяемый, проверяемый журнал» — **headline-фича проекта** — сломался, и `GET /api/admin/audit/verify` отрапортовал INVALID. На демо с нагрузкой это всплывёт.

**Усугубление:** `synchronized` — это лок **на один JVM**. Код вокруг (Spring Session JDBC, `SpringSessionBackedSessionRegistry`, комментарии «cluster-wide», «every replica») заявляет поддержку нескольких реплик. На 2+ репликах цепочка форкается **гарантированно** (см. H-3 — на деле прод single-instance, так что это латентная мина). Плюс `verifyChain` сортирует по `createdAt ASC, historyUid ASC`, а `historyUid` случайный → при равных метках времени возможны **ложные** срабатывания «битой цепочки».

**Вывод:** целостность аудита не сериализуется на уровне БД (нет `SELECT … FOR UPDATE` / выделенной last-hash строки с блокировкой / уникального индекса на `prev_hash`). Фича демонстрируема только в один поток.

## H-2. Изоляция тенантов — частичная и fail-open (рекламируется сильнее, чем есть)

**Где:** `tenant/TenantFilterAspect.java:29-42`, `domain/*` (`@Filter`), `web/TenantInterceptor.java:45-50`, `service/GatewayService.java:297`, `web/AdminController.java:91`.

Несколько независимых проблем, каждая ослабляет «жёсткую SaaS-изоляцию (audit 5.3)»:

1. **Hibernate `@Filter` НЕ применяется к `findById` / `EntityManager.find`.** Это классическая ловушка Hibernate: фильтр действует на HQL/Criteria/коллекции, но **не** на загрузку по первичному ключу. А `findById` по пользовательскому UUID есть в горячих местах:
   - `GatewayService.scanIdentityProfile` (`:297`): `identityRepository.findById(UUID.fromString(raw))` — `Identity` тенант-скоупленный, но любой авторизованный пользователь, отсканировав `IDENTITY:<любой-uuid>`, **резолвит личность из чужого тенанта** и шлёт ей уведомление (cross-tenant notification spam). UUID бизнес-карты вообще **фиксирован и задокументирован** в README (`DataSeeder:47` `aaaaaaaa-…-007`) → гарантированно перечислим.
2. **Aspect fail-open:** `catch (Exception ignored)` (`:39-41`). Если включение фильтра падает по любой причине — запрос идёт **без скоупа** (читает все тенанты). Безопасность должна fail-**closed**.
3. **Часть сущностей вообще не тенант-скоуплены:** `Interaction`, `Notification`, `Complaint`, `Event`, `Organization`, `UserSession` не реализуют `TenantScoped`. Напр. `HistoryController.myHistory` (`:77`) читает `interactionRepository.findByTargetIdentityUidOrderByCreatedAtDesc(me)` без тенант-фильтра.
4. **Фронт прямо врёт про изоляцию:** `app.js:1091` — «Показаны только пользователи **вашей организации** (изоляция тенантов)», тогда как `AdminController.users` (`:91`) делает `userRepository.findAll(pageable)` **без скоупа** (админ — глобальный super-admin) и показывает пользователей **всех** тенантов. Текст в UI противоречит фактическому поведению.

**Импакт:** реальный сквозной слив данных ограничен (имена режутся вторичными фильтрованными запросами), но модель изоляции **дырявая и fail-open**, а UI даёт ложные гарантии. Для «жёсткой изоляции» это не проходит.

## H-3. Прод работает на file-based H2 на одном диске — это структурно несовместимо с заявленной мульти-репликовой архитектурой

**Где:** `application-prod.properties:17` (`jdbc:h2:file:/app/data/ideaqr`), `render.yaml:11-46` (single web service, `plan: starter`, один диск 1 ГБ), против `SecurityConfig.java:66-81` + комментарии про «every replica», «cluster-wide».

**Что не так:** проект несёт целый слой сложности ради горизонтального масштабирования — Spring Session JDBC, `SpringSessionBackedSessionRegistry`, ревокация сессий «на всех репликах» (audit 3.7/3.9). Но **боевая СУБД — это файловый H2**, который:
- держит **эксклюзивный файловый лок** → второй инстанс просто не стартует (или, при отдельных дисках, получите split-brain с двумя разными БД);
- персистентный диск Render монтируется к **одному** инстансу — его нельзя расшарить между репликами.

То есть масштабироваться, ради чего всё затевалось, **нельзя**. А если кто-то поднимет `numInstances > 1` — прод ляжет/побьётся. Плюс H2 как боевое хранилище: слабее по конкурентности/надёжности (что прямо усиливает H-1).

**Импакт:** архитектурное противоречие + латентная мина доступности. Либо переходите на Postgres (профиль есть — `application-postgres.properties`), либо честно признайте, что это single-node, и уберите мульти-репликовые заявления.

## H-4. Цепочка владения объектом передаётся без проверки существования нового владельца

**Где:** `service/ObjectLifecycleService.java:84-143` (`transfer`), `web/ObjectAdminController.java:62-77`.

**Что не так:** `transfer(actor, objectUid, newOwnerIdentityUid, note)` не проверяет, что `newOwnerIdentityUid` соответствует **реальной** `Identity`. Можно передать объект на произвольный/несуществующий UUID:
```java
object.setOwnerIdentityUid(newOwnerIdentityUid); // никакой проверки identityRepository.existsById(...)
```
Аналогично `activate/modify/archive` не имеют предусловий, кроме `ARCHIVED`. Эндпоинт admin-only (это снижает риск), но «непрерывная цепочка владения» (заявленная фича — Расулу/продажа авто) ломается: объект получает **владельца-сироту**, которого нет в системе, и история «передан → в никуда».

> Доп. замечание: `ObjectAdminController` (в отличие от `QrAdminController`/`AdminController`/`UserAdminController`) **не** имеет `@PreAuthorize` — держится только на URL-матчере `/api/admin/**`. Defense-in-depth пробит (хоть URL и закрывает).

# 🟡 MEDIUM

## M-1. Устаревший Spring Boot 3.2.5 с известными CVE, нет сканирования зависимостей
**Где:** `pom.xml:10`.
Spring Boot **3.2.5** — апрель 2024, ~14 месяцев без обновлений. В последующих 3.2.x/3.3.x закрыты, среди прочего, **CVE-2024-38816 / CVE-2024-38819** (path traversal в статике Spring WebMVC), ряд **DoS в embedded Tomcat**, фиксы `UriComponentsBuilder`. Нет `dependency-check`/`dependabot`. Для «инвестор-стейдж MVP» зависимости должны быть свежими. → Поднять минимум до актуальной 3.2.x (лучше 3.3.x) и включить аудит зависимостей.

## M-2. Непагинированные admin/list-эндпоинты грузят таблицы целиком в память
**Где:** `web/QrAdminController.java:49-71` (`/qr/list`), `web/AdminController.java:165-202` (`/complaints`, `/events`), `service/QrService.java:141-143` (`listObjectsForAdmin` → `findAllByOrderByCreatedAtDesc`), `service/AuditService.java:87-89`.
Пагинацию добавили **только** для `/api/admin/users` (`:88`). Остальные списки (объекты, жалобы, события, глобальный журнал) читают всю таблицу. По мере роста данных — рост памяти/латентности до OOM (особенно с `-XX:+UseSerialGC` и 70% RAM). Журнал событий растёт на каждое действие.

## M-3. `GlobalExceptionHandler` отдаёт сырой `ex.getMessage()` клиенту (в т.ч. в проде)
**Где:** `web/GlobalExceptionHandler.java:61-73, 55-59`.
`server.error.include-message=never` в проде влияет только на дефолтный error-attribute Spring, но **не на `@RestControllerAdvice`**, который явно кладёт `ex.getMessage()` в тело. Утекают внутренние детали: `"Личность не найдена: <uuid>"` (`IdentityService:81`), `"Пользователь не найден: <username>"` (`UserService:157`) → перечисление/раскрытие UUID и существования. Плюс хрупкая маршрутизация `message.contains("аутентификаци")` (`:69`) для выбора 401 — сломается при смене формулировки.

## M-4. IDOR на создании жалобы (нет проверки владения взаимодействием)
**Где:** `service/ComplaintService.java:34-61`, `web/ComplaintController.java:38-50`.
`interactionRepository.findById(interactionUid)` берёт **любой** `interactionUid` без проверки, что взаимодействие принадлежит автору жалобы. Пользователь может привязать жалобу к **чужому** взаимодействию; в его собственный журнал (`auditService.record(author, interaction.getObjectUid(), …)`) утечёт чужой `objectUid`. Слабая целостность governance-объекта.

## M-5. SPA не обрабатывает 401/протухшую сессию — «всё сломалось» вместо логина
**Где:** `app.js:339-350` (`apiJson` возвращает `status`, но никто его не проверяет), множество вызовов `if (!ok) {…«Не удалось загрузить»…}`.
Когда сессия истекла (30 мин, `application.properties:56`) или админ её отозвал (audit 3.9), каждый POST/GET тихо падает, и панели показывают «Не удалось загрузить…» по всему UI. Возврат на экран логина происходит **только при ручной перезагрузке** страницы (`init→loadMe`). Нет глобального перехвата `status === 401 → renderAuth()`.

## M-6. «Рабочий режим» (working mode) для авторизации — чисто декоративный
**Где:** `service/SessionService.java:41-71`, `service/ValidationService.java:49` (`identity.getRoles()`).
Вход в рабочий режим организации меняет только поля `UserSession` (`mode/activeRole/activeOrg`). Движок политик (`ValidationService.decideAccess`) использует **постоянные** `identity.getRoles()`, а не `activeRole`. Итог: «рабочий режим» не даёт **никаких** новых прав (доступ врача работает и без него; гражданин в рабочем режиме не получает ничего). Эскалации нет — но фича подаётся как значимый контекст-свитч, а фактически пустая.

## M-7. `report` и `sos` валидируют ввод слабее, чем `scan`; обрезание по 380 символов «молча»

# 🟢 LOW

## L-1. `esc()` не нейтрализует `javascript:`/`data:` URL в ссылке карточки товара
**Где:** `app.js:2162` — `<a … href="${esc(a.url)}" …>`; `esc` (`:35-39`) экранирует только `& < > " '`. Значение `a.url` приходит из admin-создаваемого объекта (`QrService.buildDataJson:197`). `href="javascript:…"` пройдёт. Админ доверенный → Low, но это XSS-сток: валидируйте схему (`http/https`).

## L-2. Нет валидации символов в `username`/`firstName`/`lastName`
**Где:** `dto/RegistrationRequest.java:16-33` — только `@Size`, нет `@Pattern`. Принимаются HTML/JS/юникод/пробелы. Сейчас спасает **тотальный** `esc()` на выводе (проверил — дисциплина соблюдается во всех 65 `innerHTML`), но это «безопасность по удаче»: один забытый `esc()` = stored XSS (имя сканировавшего показывается **другому** пользователю в «кто меня сканировал»/«подтвердить доступ»). Добавьте allow-list символов на входе.

## L-3. «Доступ разрешён» к несуществующим объектам
**Где:** `service/ValidationService.java:44` — параметр `known` **не используется**; `RegistryClient.inferCategory:80-91`. Скан любой строки с префиксом (`PROD…`, `OBJ…`, `SERVICE…`, `RETAIL…`) → `RETAIL/GENERAL` → `APPROVED` с фабрикованной карточкой (`title = ввод пользователя`). Платформа «выдаёт доступ» к тому, чего нет в реестре. Логическая дыра/вводит в заблуждение.

## L-4. `JAVA_OPTS` из `render.yaml` молча игнорируется
**Где:** `render.yaml:21-22` задаёт `JAVA_OPTS` (`UseSerialGC`, `ExitOnOutOfMemoryError`), но `Dockerfile:31` — `ENTRYPOINT ["java","-XX:MaxRAMPercentage=75.0","-jar","app.jar"]` — **не ссылается на `$JAVA_OPTS`**. Все тщательно подобранные флаги (в т.ч. `ExitOnOutOfMemoryError`) не применяются. Плюс конфликт значений (70% в yaml vs 75% в Dockerfile).

## L-5. Postgres-профиль с дефолтным паролем `ideaqr/ideaqr`
**Где:** `application-postgres.properties:13-14` — `${DB_USERNAME:ideaqr}` / `${DB_PASSWORD:ideaqr}`. Если активировать профиль `postgres` без env-переменных, тихо используются дефолтные креды. Должно быть обязательным (без дефолта) либо fail-fast.

## L-6. Заблокированный пользователь видит «неверный логин или пароль»
**Где:** `SecurityConfig.java:153-155` (единый failureHandler) + `CustomUserDetailsService.java:40` (`accountLocked`). `LockedException` маппится в то же generic-сообщение. Пользователь не понимает, что заблокирован. (Спорно: не раскрывать статус блокировки — норм; но тогда и в UI это не отражено иначе.)

## L-7. Временный пароль возвращается в теле JSON
**Где:** `service/UserAdminService.java:161-177`, `web/UserAdminController.java:91-98`. `resetPassword` отдаёт plaintext в ответе (by-design — админ передаёт его пользователю; `mustChangePassword=true`). Для демо отметить: пароль попадёт в логи прокси/историю браузера/`X-Rate-Limit` трассировки. 8 символов из 55-символьного алфавита — приемлемо, но короче пользовательской политики (12).

## L-8. Монолитные неминифицированные ассеты + широкая `innerHTML`-поверхность
**Где:** `static/app.js` (138 КБ, один файл), `static/styles.css` (56 КБ), 65 присваиваний `innerHTML`. Нет сборки/минификации/разбиения; вся XSS-безопасность держится на ручном `esc()`. Камера запрашивает только `facingMode:environment` без перечисления камер и требует secure-context — на десктопе/http просто тост «не удалось открыть камеру» (`app.js:2322-2325`). Перфоманс/поддерживаемость/UX.

## ✅ Что сделано хорошо (чтобы было честно)

- **Нет SQL-инъекций:** только derived-queries/JPA, без конкатенации SQL.
- **Пароли:** BCrypt, нигде не сериализуются (везде явные DTO, не сущности); `User.passwordHash` не утекает.
- **Привилегии:** публичная регистрация и гость **всегда** `CITIZEN` (`UserService.register:61`), профессия с клиента игнорируется — «зарегистрироваться админом» закрыто. Спец-роли только через trusted-path/админа.
- **CSRF:** double-submit cookie + `X-XSRF-TOKEN`, `CsrfCookieFilter` материализует токен; SPA эхо-ит корректно.
- **Session fixation:** `changeSessionId()` на гостевом входе; форм-логин — дефолтная защита Spring.
- **Owner Approval Flow (P2P):** `requirePendingTarget` (`GatewayService:484-494`) реально проверяет, что запрос адресован владельцу — **здесь IDOR нет**.
- **Guest merge:** переписан на one-time merge-token (`GuestService:52-58`, `Hashing.constantTimeEquals` — `MessageDigest.isEqual`, реально constant-time) — знание UUID гостя больше не достаточно.
- **Медицина/инфраструктура:** реально гейтятся ролью + trust + рабочими часами по **серверным** часам (`ValidationService`), спуфинг времени закрыт.
- **Notification.markRead:** проверяет владельца (`NotificationService:44`) — IDOR нет.
- **Секреты:** `data/`, `.idea/`, `zakazdar/` (документы заказчика) в `.gitignore`; БД и H2-пароль не в репозитории; Docker — non-root user.
- **Аудит-чейн** концептуально верный (SHA-256 prev→entry, `verifyChain`) — ломает его только конкурентность (H-1), а не дизайн хеширования.

## 🔴 C-1 — ПОДТВЕРЖДЕНО на 100%. Rate-limiting обходится заголовком `X-Forwarded-For`

**Брутфорс логина:**

| Сценарий | Запросов | Результат |
|---|---|---|
| Постоянный IP (без `X-FF`) | 16 × `POST /login` | **10 × 401 → 6 × 429** — лимитер работает |
| Ротация `X-Forwarded-For` | 30 × `POST /login` | **30 × 401, 0 × 429** — лимита нет |
| Контрольный «взлом» | 1 × `POST /login` `admin/Admin123!`, IP `203.0.113.77` | **HTTP 200 — вход выполнен** |

Подменяя `X-Forwarded-For` на каждый запрос, я **залогинился админом**, пока «легитимный» IP уже был в бане (429). Защита от брутфорса (audit 4.9) **не существует** на практике.

**Флуд гостей (resource-exhaustion / раздувание БД):**

| Сценарий | Запросов | Результат |
|---|---|---|
| Постоянный IP | 8 × `POST /api/auth/guest` | **5 × 200 → 3 × 429** (`guestCapacity=5`) |
| Ротация `X-Forwarded-For` | 40 × `POST /api/auth/guest` | **40 / 40 × 200, 0 × 429** |

Контроль роста БД через `GET /api/admin/stats`:
```
ДО:    {"users":7,"guests":0, "qrCodes":7,  ...}
ПОСЛЕ: {"users":7,"guests":45,"qrCodes":52, ...}   ← +45 личностей и +45 QR за секунды, с одной машины
```
Каждый гость = `Identity`+`Qr`+`Assignment`+`User`+`History`. На бесплатном диске 1 ГБ (`render.yaml`) это прямой путь к отказу.

## 🟠 H-1 — ПОДТВЕРЖДЕНО и ХУЖЕ, чем в прогнозе. «Неизменяемый» журнал невалиден УЖЕ ПОСЛЕ СИДИРОВАНИЯ

Я ожидал поломку только при конкуренции. Реальность жёстче:

```
Свежая БД, НОЛЬ атак, сразу после старта:
GET /api/admin/audit/verify  ->  {"valid":false,"entriesChecked":4,"brokenAtHistoryUid":"6d068039-..."}
```
**`verifyChain()` возвращает `valid:false` сразу после загрузки** — ещё до единого скана. Воспроизвёл дважды на двух чистых БД, оба раза разрыв на записи №4. После 40 параллельных сканов (`scans:40`, `interactions:40`) журнал так и остаётся `valid:false`.

**Корневая причина (найдена в коде + подтверждена эмпирически):**
- столбец `created_at timestamp(6)` (`V1__init_baseline.sql`) — точность **микросекунды**;
- `AuditService.record` обеспечивает монотонность через `tip.getCreatedAt().plusNanos(1)` — это **9-й знак (нано), НИЖЕ микросекунды → при записи в `timestamp(6)` он теряется**;
- записи, созданные в одну микросекунду (как 7 аккаунтов в сидере), получают **одинаковый** `created_at`;
- `verifyChain` сортирует `ORDER BY createdAt ASC, historyUid ASC`, а `record` берёт tip `…DESC, historyUid DESC` → при равных метках побеждает **случайный `historyUid`**, не совпадающий с порядком хеш-связей → ложный разрыв.

То есть «защита от коллизий» (`plusNanos(1)`) — **no-op против `timestamp(6)`**. Headline-фича «неизменяемый проверяемый аудит» **не проходит собственную проверку на чистой системе**. Конкуренция — лишь дополнительный триггер.

## 🟠 H-2 — ПОДТВЕРЖДЕНО. `findById` обходит тенант-фильтр (cross-tenant resolve)

Гражданин (`citizen`, тенант PUBLIC) сканирует личность из тенанта больницы:
```
scan IDENTITY:347b2283-...(doctor, hospital tenant)  -> outcome=REVIEW  «Запрос доступа к профилю отправлен…»
scan IDENTITY:11111111-...(несуществующий UUID)      -> outcome=REJECTED «Личность по этому QR не найдена…»
```
Реальная чужетенантная личность **резолвится** (REVIEW + создаётся запрос/уведомление владельцу из другого тенанта), а несуществующая — нет. Разное поведение = `identityRepository.findById(...)` **игнорирует Hibernate `@Filter`**, как и предсказано. Cross-tenant перечисление + спам-уведомления возможны (UUID бизнес-карты вообще зафиксирован в README).

## 🟢 L-3 — ПОДТВЕРЖДЕНО. «Доступ разрешён» к несуществующему объекту
```
scan PROD-DOES-NOT-EXIST-99999  ->  outcome=APPROVED, category=RETAIL, «Объект общедоступен. Данные предоставлены.»
```
Платформа выдала APPROVED на полностью выдуманный идентификатор (инференс по префиксу `PROD`, параметр `known` не проверяется).

## 🟡 M-3 — ПОДТВЕРЖДЕНО. Сырой текст исключения уходит клиенту
```
POST /api/v2/access/<random-uuid>/confirm  ->  {"success":false,"message":"Запрос доступа не найден."}
```
`ex.getMessage()` из `@RestControllerAdvice` отдаётся как есть (в проде `server.error.include-message=never` это **не** перехватывает).
